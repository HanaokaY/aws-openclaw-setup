0. 目的と完成イメージ

できるようになること

  • ローカルPCにOpenClawを入れずに、AWS上の隔離環境（EC2）でOpenClawを動かす
  • でもOpenClawの操作はローカルPCからできる！

なぜAWS隔離が良いの？

OpenClawは"OS操作"を含む強い権限を扱うことが多く、Microsoftも「通常のワークステーションではなく、隔離環境での運用」を推奨する趣旨の注意喚起を出しています。
Microsoft - Running OpenClaw safely
もしローカルPCでプロンプトインジェクションを受けた場合、大事なデータを勝手に送信してしまう可能性があります。

2. AWS側セットアップ

2-1. AWSアカウントの作成（初心者向け）

まだAWSアカウントを持っていない場合は、まずアカウントを作成します。

  1. AWS 公式サイトから「アカウント作成」を開始
  2. メールアドレス・パスワード・AWSアカウント名を設定
  3. 連絡先情報・支払い情報（クレジットカード等）を入力（Free Tier でも入力は必要です）
  4. 本人確認（電話またはSMS）を完了すると、アカウントが有効化されます
⚠️ 重要：ルートアカウントで作業しないこと
作成直後は「ルート」という権限がすべてあるアカウントでログインした状態です。このルートアカウントで日々の作業（EC2作成・IAM操作など）をしてしまうと、誤操作や漏洩時に被害が大きくなります。AWSも「日常の作業はIAMユーザーで行う」ことを推奨しています。
→ 次の 2-2 でIAMユーザーを作成し、適切なポリシーを付与したうえで、そのユーザーで作業するようにしてください。
AWSアカウント作成完了後のコンソール画面（ルートでログインした状態）のスクリーンショット

2-2. IAMユーザーを作成し、適切なポリシーを付与する（作業はこのユーザーで行う）

上記のとおり、ルートではなくIAMユーザーで作業するため、ここで「作業用」のIAMユーザーを作成します。この手順書の以降の操作（EC2・SSM・コンソール・CLI）は、すべてこのIAMユーザーで行います。

  1. AWSコンソールにルートでログインした状態で、IAM → ユーザー → ユーザーを作成
  2. ユーザー名（例：openclaw-operator）を入力
  3. 「許可の設定」
    • 方式：ポリシーを直接アタッチ
    • ポリシー：AdministratorAccess をチェック（検証用として一通りの操作ができるようにするため。本番運用では最小権限のポリシーに変更することを推奨）
  4. 作成後、そのユーザー用のアクセスキー（プログラムによるアクセス用）を作成し、アクセスキーIDとシークレットを安全に保管。ローカルの aws configure ではこのキーを使います
  5. コンソールの操作もルートからサインアウトし、作成したIAMユーザーでサインインしてください
レクチャー後や本番運用では、最小権限のポリシーに変更するか、このユーザーを無効化・削除する運用にすると安全です。
IAM ユーザー作成画面（許可の設定で AdministratorAccess をアタッチ）のスクリーンショット

2-3. まず "リージョン" を決める（超重要）

リージョンとは？

リージョンは、AWSのサーバーが実際に置かれている「場所」の単位です。東京、バージニア（米東部）、シドニーなど、世界中に複数のリージョンがあり、それぞれに名前（リージョンコード）が付いています。

  • 例：東京 → ap-northeast-1、大阪 → ap-northeast-3、シドニー → ap-southeast-2
  • EC2やS3などのリソースは「どのリージョンに作るか」を選ぶ必要があり、リージョンが違えば別のリソースとして扱われます（東京のEC2とシドニーのEC2は別物です）。
  • コンソール（ブラウザ）で表示しているリージョンと、ローカルPCのAWS CLIで指定しているリージョンが一致していないと、「そのリージョンにはそのインスタンスがありません」となり、この手順書で後から使うSSMのポートフォワードで TargetNotConnected というエラーが出やすくなります。

そのため、最初に「どこのリージョンで作業するか」を1つ決めて、コンソールもCLIもそのリージョンで統一することがとても重要です。

手順：リージョンを1つ決めて統一する

  • EC2を作るリージョン：AWSコンソール右上のリージョン表示をクリックし、使うリージョンを選ぶ（例：東京 ap-northeast-1 / シドニー ap-southeast-2）。このリージョンでEC2やIAMの操作を行います。
  • ローカルAWS CLIのリージョン：後述の aws configure で設定する region を、上で選んだリージョンと同じにします。
AWSコンソール右上のリージョン選択のスクリーンショット

2-4. EC2にSSMで入るためのロールを作成（必須）

ロールとは？

ロール（IAMロール）は、AWSのリソースに「どのサービスが」「どんな権限で」アクセスしてよいかをまとめた権限セットです。

  • 人間が使うのは「IAMユーザー」ですが、EC2インスタンスなどのAWSリソース自体にも「AWSの中の別サービスにアクセスする権限」が必要になることがあります。
  • 今回のケースでは、EC2がSystems Manager（SSM）と通信できるようにするための権限が必要です。
  • その「EC2に付ける権限のかたまり」が IAMロール であり、EC2にロールをアタッチすることで、「このEC2はSSMとやり取りしてよい」ということをAWS側に伝えます。

このロールがないと、EC2がSSMと接続できず、ブラウザからのポートフォワードやSession Manager接続ができません。そのため、このステップは必須です。セキュリティのためですね！

手順：SSM接続用ロールを作成する

  1. AWSコンソールで IAM を開き、左メニューから ロール → ロールを作成 を選択
  2. 「信頼されたエンティティの種類」で AWSサービス を選び、ユースケースに EC2 を選択
  3. 「許可ポリシーをアタッチ」で、検索欄に AmazonSSMManagedInstanceCore と入力し、そのポリシーにチェックを入れる（SSMの基本的な権限のセットです）
    AWS ドキュメント - SSM Agent
  4. ロール名を例として EC2-SSM-Role とし、内容を確認して作成
IAM ロール作成（信頼ポリシー・許可ポリシー）のスクリーンショット

2-5. EC2を作成（OpenClaw用）

EC2とは？（何を作るステップか）

EC2（Elastic Compute Cloud）は、AWSが提供する「クラウド上の仮想サーバー」です。自分のPCとは別に、AWS上にもう1台パソコンを借りるイメージです。

  • OS（ここでは Amazon Linux 2023）を選んで起動し、そこでソフトウェア（今回だと OpenClaw）を動かします。
  • ブラウザで開く Dashboard の裏側では、このEC2上で OpenClaw の Gateway プロセスが動いています。
  • この手順では、OpenClaw専用の一時的なEC2 を1台だけ立てて、検証が終わったらTerminate（削除）する前提です。

ここで作るEC2の性能（インスタンスタイプ）やディスクサイズによって、OpenClawの動きやすさやコストが変わります。

借りるパソコンのスペックを選ぶ感じです！

推奨インスタンス

t3.small（OpenClawはそれなりに重いので micro だとハマりやすい）

作成手順（EC2 起動ウィザード）

  1. EC2 → インスタンスを起動
  2. AMI：Amazon Linux 2023
  3. インスタンスタイプ：t3.small
  4. キーペア：作らなくてOK（SSHしないので）
  5. ネットワーク（セキュリティグループ）
    • inbound から SSH(22) を削除
    • 結果：インバウンドルールなし（これが正解）
  6. IAMインスタンスプロファイル：EC2-SSM-Role を選択
  7. ストレージ：20GB程度でOK
  8. タグ（任意・消し忘れ防止）
    • Name = openclaw-lab
    • Owner = （名前）
    • AutoDeleteAt = （日時）
EC2 インスタンス起動ウィザード（AMI・インスタンスタイプ・SG・IAM）のスクリーンショット

2-6. SSMでEC2に接続（SSH不要）

  1. EC2 → インスタンス選択 → 接続
  2. Session Manager タブ → 接続

もし接続できないとき
EC2側にSSM Agentが必要です。Amazon Linux 2023なら基本入っていますが、確認コマンド：

sudo systemctl status amazon-ssm-agent --no-pager
EC2「接続」→ Session Manager タブのスクリーンショット

3. EC2上にOpenClawをインストール（公式インストーラー推奨）

チーム初心者には 公式インストーラーが最短です。OpenClaw公式は「インストーラーでNode検出〜導入〜オンボードまで」を推奨しています。（Install - OpenClaw Docs）

3-1. 必要パッケージ

sudo dnf update -y
sudo dnf install -y curl git

3-2. OpenClaw公式インストーラーで導入

curl -fsSL https://openclaw.ai/install.sh | bash
✅ もし「root権限なしで入れたい」場合は install-cli.sh のほうが合うケースがあります（Installer Internals - OpenClaw Docs）

4. OpenAI APIキーを設定（OpenClawがモデルを使えるようにする）

OpenClawはOpenAIをプロバイダとして使えます。（OpenAI - OpenClaw Docs）

4-1. まずはAPIキーを用意する（考え方）

⚠️ キーはチャットや画面共有で読み上げない・貼らないでください。GitHubなどのソースコード管理にも絶対にコミットしないようにします。

OpenClawでOpenAIを使うには、OpenAIのAPIキーが1つ必要です。
この手順書では、OpenClawのオンボード（設定ウィザード）でAPIキーを直接入力する方式を前提にします。

4-2. OpenClawのオンボード/設定ウィザード

OpenClawには configure または onboard というウィザードがあり、対話形式でAPIキー入力や初期設定を誘導してくれます。（Onboarding Wizard Reference - OpenClaw Docs）

EC2上で次を実行すると、ウィザードが始まります。

openclaw configure

（openclaw onboard でも同様の流れです）

ウィザードで聞かれる主な項目（何を選べばよいか）

実行すると、順番に次のような質問や選択肢が出ます。この手順書では「OpenAI を使って OpenClaw を動かす」前提で進めるので、次のように答えれば問題ありません。

  • プロバイダ／モデルの選択 … 利用するAIプロバイダを選びます。ここでは OpenAI を選びます。
  • APIキーの入力 … OpenAIのAPIキー（sk-... で始まる文字列）を、表示されたプロンプトにコピー＆ペーストで入力します。入力中は画面に表示されないことがありますが、そのまま入力してEnterでOKです。
  • デフォルトモデル … 使用するモデル（例：gpt-4o, gpt-4o-mini など）を選ぶか、候補から選びます。よく分からなければ表示されているデフォルトのままで大丈夫です。
  • ワークスペース … エージェント用のファイルを置く場所を聞かれることがあります。そのまま既定のパスで問題ありません。
  • Gateway（ポート・認証など） … 後で openclaw --dev gateway run を使う場合は、ここは既定のままでかまいません。
  • その他（スキル・デーモン・チャネルなど） … スキルの導入やバックグラウンドサービス、チャット連携などが聞かれる場合があります。まずは「スキップ」や既定の選択で進めてよいです。必要になったらあとから設定できます。

途中で分からなくなったら、その項目で Enter を押すとよく使われる既定値が選ばれることが多いので、まずは Enter で進めてみてください。ウィザードが終わると、OpenAI のAPIキーやモデル設定が保存され、次のステップ（Gateway 起動）に進めます。

openclaw configure / onboard の画面（プロバイダ選択・APIキー入力・モデル選択など）のスクリーンショット

5. Gateway（Dashboard付き）を起動する

OpenClawのGatewayは、WebSocket制御/RPC + HTTP API + Control UI などを1プロセスで提供する設計です。（Gateway Runbook - OpenClaw Docs）

5-1. "devプロファイル" で起動（チーム検証に便利）

--dev は状態を ~/.openclaw-dev に隔離し、ポートもdev用に分離します（例：19001）。

openclaw --dev gateway run
既に何かがポートを掴んでいたら --force が使える場合があります（helpにも例が出ます）

5-2. DashboardのURLを確認（これが確実）

openclaw --dev dashboard

ここで例えば Dashboard URL: http://127.0.0.1:19001/ のように出ます。

gateway run と dashboard コマンドの出力のスクリーンショット

6. ローカルPCからSSMポートフォワードでDashboardを開く

SSMポートフォワードは portNumber（EC2側）と localPortNumber（ローカル側）で指定します。（セッションを開始する - AWS Systems Manager）

6-1. ローカルMacにAWS CLIを入れる

brew install awscli
aws --version

Session Manager plugin が必要な場合：

brew install session-manager-plugin
session-manager-plugin --version

6-2. aws configure（アクセスキー方式）

IAMユーザーでアクセスキーを作り、ローカルに設定：

aws configure
  • region：EC2を作ったリージョンと同じ（例：ap-northeast-1 or ap-southeast-2）
  • output：json

6-3. ポートフォワード開始（例：19001）

EC2インスタンスID（i-...）を指定します。

aws ssm start-session \
  --region <EC2のリージョン> \
  --target i-xxxxxxxxxxxxxxxxx \
  --document-name AWS-StartPortForwardingSession \
  --parameters '{"portNumber":["19001"],"localPortNumber":["19001"]}'

この状態でターミナルは「待機」になります（正常）。

aws ssm start-session でポートフォワードが「待機」になっているターミナルのスクリーンショット

6-4. ローカルブラウザで開く

http://localhost:19001/
ブラウザで OpenClaw Dashboard が表示されているスクリーンショット

7. デモ（おすすめ：nginxインストール＆起動）

7-1. Dashboardから指示（例）

チャット欄に下を貼る：

Install nginx on this machine, start the service, then show the service status and nginx version.
Use minimal commands.

7-2. "勝手にOS操作"を安全に見せるコツ：Exec approvals

OpenClawは危険操作を防ぐため、実行に承認を要求する仕組み（approvals）があります。（Exec Tool - OpenClaw）
デモでは「承認が必要な設計＝安全寄り」を見せると説得力が上がります。

  • 安全寄り（推奨）：承認が出たらDashboard/CLIで approve
  • デモ速度優先：自動承認（ただし危険なので社内検証では注意）

7-3. 人間が最後に確認（締めがキレイ）

EC2側（SSM接続）で：

sudo systemctl status nginx --no-pager
nginx -v
Dashboardでnginxインストール指示と承認フローのスクリーンショット

8. 終了時：コストを残さない片付け（最重要）

8-1. EC2をTerminate（削除）

EC2 → ボリューム → 残っていたら削除

EC2 → Elastic IP → あれば解放（今回は通常作ってないはず）

9. つまずき集（今回ハマりやすいところ）

現象 原因・対処
A) TargetNotConnected AWS CLIのリージョンとEC2のリージョン不一致が最頻出。--region を明示して合わせる。
B) ブラウザで unauthorized それは「Dashboard」じゃなく「browser control（auth=token）」に当たってる可能性大。openclaw --dev dashboard で 正しいURLとポートを必ず確認。
C) ポートフォワードが「Connection to destination port failed」 EC2側でそのポートがLISTENしてない。openclaw --dev gateway run を起動しているか確認。ss -ltnp | grep 19001 で待受確認。
D) コストがすぐ反映されない 請求/クレジットは遅延反映があり得る。新規クレジットは条件で増えることもある。
